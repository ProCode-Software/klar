# Go Style Guide

> **Disclaimer:** This style guide was generated by AI. These are just the basics and have been edited for accuracy. You should still read a variety of areas of the codebase to understand the conventions.

This guide provides a comprehensive overview of the Go coding conventions used in the Klar repository.

## 1. General Formatting & Spacing

### Line Length

- **Target**: 90 columns for code and doc comments.
- **Limit**: Up to 94 columns (and sometimes longer for function signatures) is acceptable if it's not worth breaking.

### Formatting Tool

- Always use **`gofumpt`** (a stricter version of `gofmt`).
- Ensure `GOEXPERIMENT=jsonv2` is set when running tools that support it.

### Newlines & Grouping

- Use a single blank line between functions and methods.
- **Do not** add unnecessary newlines between short, related statements inside functions.
- Don't always add newlines before returns.
- Use blank lines within functions ONLY to group distinct logical blocks or separate initialization from core logic.

```go
// Correct spacing: No extra lines between related short assignments and checks.
func (p *Parser) Advance() lexer.Token {
	tok := p.Curr()
	if tok.Kind == lexer.EOF {
		return tok
	}
	p.Index++
	return tok
}

// Correct grouping: One blank line between different functions.
func (p *Parser) Curr() lexer.Token {
	return p.Tokens[p.Index]
}

// For short functions
func (p *Parser) CurrKind() lexer.TokenType { return p.Curr().Kind }
```

## 2. Code Organization

### File Layout

1. `package` declaration.
2. `import` statements (grouped and sorted by `gofumpt`).
3. Core types and constants (at the top of the file).
4. Factory functions (e.g., `New...`).
5. Methods and functions, ordered top-down by call level.

### Sectioning

Large files are divided into logical sections using comment headers. The number of `=` characters don't match the title length. Section doesn't have to match -- just type a few.

```go
// Basic Literals
// =======

// Accessors & Collections
// ======

// To implement [reporter.Error].
// ======

// Assignments & Destructuring
// =======
```

### Type and Variable Grouping

Group 3+ related variables or types using a block:

```go
var (
	commands = klarcmd.KlarCommands
	aliases  = klarcmd.KlarCommandAliases
	profiler prof
)

type (
	StringFragment interface{ StringFrag() }
	EscapeFragment struct{ Value StringEscape }
	TextFragment   = lexer.TextFragment
)
```

## 3. Naming Conventions

### Exported vs. Unexported

- **Exported**: PascalCase (e.g., `Parser`, `NewChecker`).
- **Unexported**: camelCase (e.g., `handlePanic`, `listCastTokens`).

### Short Receivers & Variables

- Use short (1-3 letter) receiver names: `p` for `Parser`, `l` for `Lexer`, `c` for `Checker`.
- Use concise local variables: `tok` for `Token`, `id` for `Identifier`, `err` for `error`.
- In `range` loops, prefer meaningful but short names: `k, v` or `name, program`.

### Fields

- Name fields clearly based on their data (e.g., `Identifier`, `BaseNode`, `Operator`).
- Avoid redundant names (e.g., `ParserIndex` inside `Parser` struct should just be `Index`).

## 4. Types & Declarations

### Embedding for Common Logic

- Use embedding for shared metadata, such as `BaseNode` for AST nodes to provide range information.

```go
type BaseNode struct{ Range ranges.Range }

type Symbol struct {
	BaseNode
	Identifier string
}
```

### Type Aliases

- Use type aliases sometimes, especially when bridging packages:

```go
type Error = klarerrs.Error
type CommentType = lexer.TokenType
```

### Interfaces

- Interface names often describe a capability (e.g., `Node`, `Expression`, `Statement`).
- Keep interfaces small and focused.

## 5. Functions & Methods

### Top-Down Ordering

Order methods by their level of abstraction and call order. High-level methods appear before the lower-level helper methods they call.

```go
func NewChecker(mod *Module, opts *Options) *Checker {
	c := &Checker{}
	c.Init(mod, opts)
	return c
}

func (c *Checker) Init(mod *Module, opts *Options) {
	if opts == nil {
		opts = &Options{}
	}
	c.rootContext = mod.Context
	c.module = mod
	c.Programs = mod.Programs
	c.Options = opts
	c.moduleDecls = make(map[*Object]*DeclarationInfo)
	c.loadInternalModules()
}
```

## 6. Error Handling

### Klarerrs Package

- Use the internal `klarerrs` package for all compiler-related errors.
- Use the fluent API to build descriptive errors with highlights and hints.

```go
err := klarerrs.Token(klarerrs.ErrChainedNotEqual, p.Curr())
err.Hint("In 'a != b != c', 'a' could still be equal to 'c'...")
p.ErrorLabelled(err, "Can't chain '!='")
```

## 7. Comments & Documentation

### Doc Comments

- Start with the name of the symbol being documented.
- Target 90 columns.
- Do not use a period for single-sentence comments unless they are long.
- Use square brackets for symbol cross-references: `[lexer.Token]`.

```go
// A Parser parses lexer tokens into an abstract syntax tree (AST).
type Parser struct { ... }

// Curr return the [lexer.Token] at the current parser index.
func (p *Parser) Curr() lexer.Token { ... }
```

### Internal Comments

- Use `// TODO:` for missing features or improvements. Messages are usually lowercase
- Use comments to explain complex logic or syntax features being parsed.
- Comments usually don't end in periods, unless they're long or multiple sentences.

## 8. Go Language Features

### Latest Syntax

- Use **`any`** instead of `interface{}`.
- Use **`range`**
- Use **`min()`**, **`max()`**, and **`cmp.Or()`** where appropriate.
- Use **`new(expr)`** for allocating and initializing pointers.

### Standard Library

- Favor `slices` and `maps` packages over manual implementations.
- Use `strings.Builder` for efficient string concatenation.
- **ProTip:** For sorting keys in a map, use the following one-liner:

```go
keys := slices.Sorted(maps.Keys(m)) // OMG WOW!!!
```

## 9. Testing & Benchmarking

### Benchmarks

- Use `(*testing.B).Loop()` instead of `b.N` for more accurate benchmarks.

```go
func BenchmarkExample(b *testing.B) {
	for b.Loop() {
		// code to benchmark
	}
}
```

### Test Organization

- Use `t.Run` to group subtests.
- Use a `setup` helper if common initialization is required.
